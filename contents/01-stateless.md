# なぜサーバは「状態を持たない」のか — ステートレス設計

> **前提知識:** `HttpSession` でログイン状態を管理した経験がある

---

## 1. まず、Servlet で何をやっていたか思い出そう

```java
// ログイン処理
HttpSession session = request.getSession();
session.setAttribute("userId", user.getId());
session.setAttribute("role", "admin");

// 別のページで認証チェック
HttpSession session = request.getSession(false);
if (session == null || session.getAttribute("userId") == null) {
    response.sendRedirect("/login");
}
```

これは **ステートフル（Stateful）** な設計だ。  
サーバが「このユーザーはログイン済み」という **状態をメモリに保持** している。

---

## 2. ステートフルの何が問題なのか

### 問題 1: サーバを増やせない

ユーザーが増えて、サーバを2台に増やしたとする。

```
ユーザーA → [ロードバランサ] → サーバ1 (セッションあり ✅)
ユーザーA → [ロードバランサ] → サーバ2 (セッションなし ❌ → ログイン画面に戻される)
```

`HttpSession` はサーバ1のメモリにしか存在しない。  
リクエストがサーバ2に振られた瞬間、「誰だお前は」と言われる。

**回避策はあるが、どれも面倒：**
- スティッキーセッション（同じユーザーを常に同じサーバへ → 負荷が偏る）
- セッションレプリケーション（全サーバ間でセッションを同期 → ネットワーク負荷）
- 外部セッションストア（Redis にセッション保存 → 管理対象が増える）

### 問題 2: サーバレスと相性が悪い

Cloudflare Workers のようなサーバレス環境では、リクエストごとに関数が起動する。  
**前のリクエストのメモリは残っていない。** セッションを保持する場所がそもそも存在しない。

```
リクエスト1 → [Worker 起動] → 処理 → [破棄]
リクエスト2 → [Worker 起動] → 処理 → [破棄]  ← 前回のメモリはない
```

### 問題 3: モバイルアプリ・外部連携ができない

Web ブラウザの Cookie に依存する `HttpSession` は、  
モバイルアプリや他社の API から使おうとすると途端に面倒になる。

---

## 3. ステートレスとは何か

**「サーバは一切の状態を持たない。必要な情報はすべてリクエストに含める」** という設計原則。

```
【ステートフル】
  クライアント: 「データをください」
  サーバ:       「（メモリを見て）お前はAさんだな。はい、Aさんのデータ」

【ステートレス】
  クライアント: 「私はAです。証明書はこれです。データをください」
  サーバ:       「（証明書を検証して）OK、Aさんのデータだな。はい」
```

サーバは「誰がログインしているか」を覚えない。  
代わりに、**クライアントが毎回「自分が誰か」を証明する**。

---

## 4. JWT（JSON Web Token）— ステートレス認証の仕組み

### 4.1 ログイン時

```
クライアント                              サーバ
    │                                       │
    │─── POST /login {email, password} ────→│
    │                                       │── パスワード検証
    │                                       │── JWT を生成（署名付き）
    │←── { token: "eyJhbGci..." } ─────────│
    │                                       │
    │  ※ サーバはこのトークンを保存しない     │
```

### 4.2 その後のリクエスト

```
クライアント                              サーバ
    │                                       │
    │─── GET /api/users ───────────────────→│
    │    Authorization: Bearer eyJhbGci...  │
    │                                       │── 署名を検証（DBアクセス不要）
    │                                       │── ペイロードから userId を取得
    │←── { users: [...] } ─────────────────│
```

### 4.3 JWT の中身

JWT は3つのパートからなる（Base64エンコードされているだけで、暗号化ではない）：

```
eyJhbGciOiJIUzI1NiJ9.eyJ1c2VySWQiOjEsInJvbGUiOiJhZG1pbiIsImV4cCI6MTcxOH0.xxxxx
├── ヘッダー ──────────┤├── ペイロード ──────────────────────────────────┤├─ 署名 ─┤
```

```json
// ペイロード（デコードすると）
{
  "userId": 1,
  "role": "admin",
  "exp": 1718000000   // 有効期限（Unix時間）
}
```

**ポイント:** サーバは署名を検証するだけ。データベースやメモリを見に行く必要がない。  
だからどのサーバ、どの Worker が処理しても、同じ結果が返せる。

## 5. Servlet との比較まとめ

```
【HttpSession（ステートフル）】

  ブラウザ ──Cookie: JSESSIONID=abc123──→ サーバ
                                           │
                                           ↓
                                    メモリ上のセッション
                                    ┌─────────────────┐
                                    │ abc123:          │
                                    │   userId = 1     │
                                    │   role = "admin" │
                                    └─────────────────┘
  問題: このメモリは「このサーバ」にしかない


【JWT（ステートレス）】

  ブラウザ ──Authorization: Bearer eyJ...──→ サーバ（どれでもOK）
                                              │
                                              ↓
                                      署名を数学的に検証
                                      ペイロードを読むだけ
                                      （メモリもDBも不要）
```

## 6. 「でもログアウトはどうするの？」

良い質問。ステートレスの弱点の一つ。

- **HttpSession**: `session.invalidate()` で即座に無効化できる
- **JWT**: サーバ側に情報がないので「このトークンを無効にする」が難しい

**よくある解決策：**

| 方法 | 説明 | トレードオフ |
|------|------|-------------|
| 短い有効期限 | JWT を15分〜1時間で失効させる | ユーザーが頻繁に再ログインする必要がある |
| リフレッシュトークン | 短命のアクセストークン + 長命のリフレッシュトークンの2本立て | 実装が少し複雑になる |
| ブラックリスト | 無効化したトークンIDをDBに記録 | 完全なステートレスではなくなる |

現実の多くのアプリは **「短い有効期限 + リフレッシュトークン」** の組み合わせを採用している。

## 7. REST の原則としてのステートレス

ステートレスは JWT のためだけの概念ではない。**REST (Representational State Transfer)** の基本原則の一つ。

REST の制約：
1. **ステートレス** — 各リクエストは自己完結している
2. **統一インターフェース** — URL + HTTP メソッドでリソースを操作する
3. **クライアント・サーバ分離** — 画面の描画とデータの管理を分ける

Servlet/JSP 時代は、フォームを POST して画面ごとリダイレクトしていた。  
これは REST の「クライアント・サーバ分離」に反している。  
このテンプレートでは、**Backend は JSON API に徹し**、画面はフロントエンドの仕事。

---

## 8. Hono での実装イメージ

```typescript
// Backend: 認証ミドルウェア
import { jwt } from 'hono/jwt';

// この1行で、以降のルートは JWT 検証が必須になる
app.use('/api/*', jwt({ secret: 'your-secret-key' }));

// ハンドラーでは payload を取り出すだけ
app.get('/api/me', (c) => {
  const payload = c.get('jwtPayload');
  return c.json({ userId: payload.userId, role: payload.role });
});
```

Servlet で書いていた `Filter` + `HttpSession` チェックの数十行が、**ミドルウェア1行**に置き換わる。

---

## まとめ

| 概念 | Servlet (ステートフル) | このテンプレート (ステートレス) |
|------|----------------------|-------------------------------|
| 認証情報の保存場所 | サーバのメモリ | クライアント（JWT） |
| スケーラビリティ | サーバ間のセッション共有が必要 | どのサーバでも処理可能 |
| サーバレス対応 | ❌ 不可能 | ✅ 自然に対応 |
| ログアウト | `session.invalidate()` で簡単 | トークン失効戦略が必要 |
| API 連携 | Cookie 依存で面倒 | `Authorization` ヘッダーで統一 |

**覚えておくこと：** ステートレスは「セッションを使わない不便な設計」ではなく、  
「どこでも、誰でも、同じように処理できる」ためのスケーラブルな設計選択。
